0 Project Overview

The goal is to build a polished, responsive full-stack product on the Reno Stack — a modern starter that decouples a Vite/React front-end from a Hono back-end and connects them with type-safe RPC. Auth may be mocked, and dummy data is acceptable, so emphasis stays on UX quality, clean architecture, complete data-handling (search/filter/sort/pagination), and a single shared type surface across the whole app.

🛬 Marketing

A landing page that sells the product: hero, features, stats, and a call-to-action.

🔐 Auth Flow

Sign in, sign up, forgot & reset password via Better-Auth (or mocked).

📊 Dashboard

Metric cards plus a feature-rich device table with all data-handling controls.

🔎 Details

A rich device profile with warranty timeline and notes (route or side panel).

🛠 API

Hono routes serving device lists & details, consumed through Hono RPC client.

🚀 Ship

Deployed live, recorded demo, and a README explaining setup and decisions.

Why Reno fits this brief. The assignment explicitly wants device listings and details endpoints with server-side data handling — exactly the shape Reno's Hono-RPC + React Query layer was built for. SEO doesn't matter here (it's an authed app behind a marketing page), which is precisely the trade-off Reno optimizes for: speed and simplicity over SSR/SEO.

1 The Reno Stack

A pnpm monorepo that pairs a React/Vite SPA with a Hono server, wired together by type-safe RPC. Reference: github.com/reno-stack/reno-stack.

Front-End (apps/web)

React + Vite TanStack Router (file-based) TanStack Query Tailwind CSS Shadcn UI Hono RPC client

File-based routing, Shadcn components added via pnpm ui-add, and React Query bound to the API through Reno's Hono helpers.

Back-End (apps/server)

Hono Drizzle ORM PostgreSQL Better-Auth Zod

Hono routes generated with pnpm create:route; Drizzle schema pushed with pnpm db:push; Better-Auth handles sessions.

Shared (packages/*)

packages/validators packages/ui Zod schemas TypeScript everywhere pnpm workspaces

One source of truth for Zod validators and shared Shadcn UI, imported by both apps — no type duplication across the boundary.

2 End-to-End Type Safety — the core of this plan

Reno's whole point is that a type defined once flows unbroken from the database to the rendered table cell. There is no hand-written API client and no DTO drift.

🗄 Drizzle schema→ 🧬 Inferred row types→ ✅ Zod validators (packages/validators)→ 🚦 Hono route + RPC→ 🔗 Typed client (hc)→ ⚛️ React Query hook→ 🖥 UI
A change to a column or query param surfaces as a TypeScript error at every consuming call site — compile-time, not runtime.

How the link is made

  • Drizzle infers row types — InferSelectModel<typeof devices> gives the canonical Device type straight from the table definition.
  • Zod validators are shared — query params (search/filter/sort/page) and bodies live in packages/validators, imported by both the Hono route (to validate) and the web app (to type forms).
  • Hono RPC exports the app type — export type AppType = typeof app; the client is built with hc<AppType>(baseUrl), so request shapes and response JSON are fully typed.
  • Reno's React Query helpers — createHonoQueryOptions and createHonoMutationOptions wrap the RPC client so hooks like useDevices() infer data, params, and errors automatically.

Illustrative wiring

// packages/validators/devices.ts  — single source of truth
export const deviceQuerySchema = z.object({
  search:   z.string().optional(),
  status:   z.enum(["active","expiring_soon","expired"]).optional(),
  brand:    z.string().optional(),
  sort:     z.enum(["name","purchaseDate","warrantyExpiry","status"]).default("warrantyExpiry"),
  order:    z.enum(["asc","desc"]).default("asc"),
  page:     z.coerce.number().int().min(1).default(1),
  pageSize: z.coerce.number().int().min(1).max(100).default(10),
});
export type DeviceQuery = z.infer<typeof deviceQuerySchema>;

// apps/server/src/routes/devices.ts  — Hono route, validated + typed
const route = new Hono()
  .get("/", zValidator("query", deviceQuerySchema), async (c) => {
    const params = c.req.valid("query");          // DeviceQuery, typed
    const result = await listDevices(params);     // Drizzle query
    return c.json(result);                         // { data: Device[], meta }
  })
  .get("/:id", async (c) => c.json(await getDevice(c.req.param("id"))));
export type DevicesType = typeof route;

// apps/web/src/features/devices/queries.ts  — typed client + hook
export const devicesQuery = (params: DeviceQuery) =>
  createHonoQueryOptions(client.api.devices.$get, { query: params });
//  ⤷ data is inferred as { data: Device[]; meta: PageMeta } — no manual typing
Payoff for the demo. Rename warrantyExpiry in the Drizzle schema and the table column, the sort enum, and the API all light up red until fixed. That's a strong, concrete story to narrate in the submission video.

Data flow

  • Route component → calls a typed hook (useQuery(devicesQuery(params)))
  • Hono RPC client → typed request to the Hono server; TanStack Query caches by key, exposes isPending / isError / data
  • Hono route → Zod-validates query, runs the Drizzle search→filter→sort→paginate pipeline, returns { data, meta }
  • UI → skeletons / empty / table; table state lives in TanStack Router search params so views are shareable

4 Drizzle Schema & Seed

A single core entity — the device — plus warranty events, with status derived from dates.

// apps/server/src/db/schema.ts
export const devices = pgTable("devices", {
  id:             uuid("id").primaryKey().defaultRandom(),
  name:           text("name").notNull(),
  brand:          text("brand").notNull(),
  category:       text("category").notNull(),
  model:          text("model").notNull(),
  serialNumber:   text("serial_number").notNull(),
  purchaseDate:   timestamp("purchase_date").notNull(),
  purchasePrice:  integer("purchase_price").notNull(),
  retailer:       text("retailer").notNull(),
  warrantyMonths: integer("warranty_months").notNull(),
  warrantyExpiry: timestamp("warranty_expiry").notNull(),  // stored, kept in sync
  notes:          text("notes").default(""),
  createdAt:      timestamp("created_at").defaultNow(),
});

export const warrantyEvents = pgTable("warranty_events", {
  id:          uuid("id").primaryKey().defaultRandom(),
  deviceId:    uuid("device_id").references(() => devices.id).notNull(),
  date:        timestamp("date").notNull(),
  type:        text("type").notNull(),   // purchase|registered|claim|repair|extended|expired
  title:       text("title").notNull(),
  description: text("description"),
});

export type Device = InferSelectModel<typeof devices>;        // ← used everywhere
export type WarrantyEvent = InferSelectModel<typeof warrantyEvents>;

Status derivation rule (shared helper)

ConditionStatusBadge
expiry < todayexpired● Expired
0 ≤ daysLeft ≤ 30expiring_soon● Expiring soon
daysLeft > 30active● Active

Status is computed (not stored) by a helper in packages/validators so the table, the dashboard metrics, and the details view share one definition. The 30-day window powers the Expiring Soon metric. Seed ~250 devices with Faker via pnpm db:seed.

Validation & guarding

  • Schema-driven — Zod schemas in packages/validators wired through React Hook Form for field + submit errors.
  • States covered — invalid email, weak/short password, mismatched confirm, required fields, submit loading, server error ("Invalid credentials").
  • Better-Auth — provides the session, typed useSession(), and server middleware to protect Hono routes.
  • Route guard — TanStack Router beforeLoad redirects unauthenticated users away from /dashboard and /devices/*.
  • Mock fallback — if you'd rather not run real auth, the brief allows mocking: stub the Better-Auth client to accept any credentials and short-circuit the guard.

9 Back-End / API Layer (Hono)

Hono routes serving device data with all data-handling done server-side in Drizzle. Generate each with pnpm create:route.

Endpoints

MethodPathPurpose
GET/api/devicesPaginated list with search, filter, sort
GET/api/devices/:idFull device profile incl. timeline & notes
PATCH/api/devices/:id/notesUpdate notes (typed mutation)
GET/api/statsSummary metrics for dashboard cards
GET/api/metaDistinct brands/categories for filter dropdowns
GET/healthHealth check for deployment

List query contract

GET /api/devices
  ?search=macbook        // matches name, brand, model, serial
  &status=active         // active | expiring_soon | expired
  &brand=Apple
  &sort=warrantyExpiry   // name | purchaseDate | warrantyExpiry | status
  &order=asc             // asc | desc
  &page=1
  &pageSize=10

Response:  { "data": Device[], "meta": { page, pageSize, total, totalPages } }
// validated by deviceQuerySchema; response type inferred by the Hono RPC client

Drizzle service pipeline (order matters)

  • 1 · Search — ilike across indexed text columns (name, brand, model, serial).
  • 2 · Filter — where by status (computed range on expiry) and brand/category.
  • 3 · Sort — orderBy requested column + direction, stable secondary key (id).
  • 4 · Paginate — limit/offset; count() the filtered set for meta.
  • Validation — zValidator("query", deviceQuerySchema) coerces & bounds params (pageSize ≤ 100), 400 on bad input.
  • Errors — Hono onError returns { error: { message, code } }; 404 for unknown ids.
  • Seed — pnpm db:seed inserts ~250 Faker devices + timelines so demos look real.
  • CORS — Hono CORS middleware scoped to the web origin; Better-Auth middleware protects device routes.
